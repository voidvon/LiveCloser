## Project Overview

A full-stack MVP for an AI-powered sales voice agent using LiveKit RTC. The system has three main processes:
1. **KB Service** — FastAPI REST API (port 8001) for knowledge base, product catalog, and settings management
2. **Agent** — Python LiveKit agent that handles voice/text conversations with RAG and function tools
3. **Frontend** — Next.js 15 app that connects via LiveKit WebRTC and proxies KB API calls

## Common Commands

### Starting Development

```bash
# Start all components (recommended)
./scripts/dev.sh

# Start individual components
./scripts/dev.sh backend    # Agent + LiveKit server
./scripts/dev.sh frontend   # Next.js only (port 3000)
./scripts/dev.sh kb         # KB service only (port 8001)
./scripts/dev.sh console    # Text-only console mode (no voice/audio)
```

### Manual Component Start

```bash
# KB Service
UV_CACHE_DIR=.uv-cache uv run uvicorn src.kb_server:app --host 127.0.0.1 --port 8001 --reload

# Agent (voice)
UV_CACHE_DIR=.uv-cache uv run python src/agent.py dev

# Agent (text console, no audio deps required)
UV_CACHE_DIR=.uv-cache uv run python src/agent.py console --text

# Frontend
cd frontend && pnpm dev
```

## Architecture

### Request Flow

```
User (browser) ──WebRTC──► LiveKit Server ──dispatch──► Python Agent
     │                                                        │
     └──HTTP──► Next.js API Routes ──proxy──► KB Service (FastAPI)
```

- Frontend never calls KB service directly — all requests go through `/api/kb/*` and `/api/chat/*` Next.js proxy routes
- LiveKit tokens are generated by `/api/token` route using server-side credentials
- Agent connects to LiveKit server and handles voice/text from participants

### Python Backend Structure

- **`src/agent.py`** — `SalesAgent` class (extends `livekit-agents` `Agent`). Entry point for agent dispatch. Handles session lifecycle, RAG context injection, idle timeout management, and exposes two function tools: `search_knowledge_base()` and `search_products()`.
- **`src/kb_server.py`** — FastAPI app. Provides CRUD for knowledge bases, products, agent/model profiles, file upload/indexing jobs, and conversation history.
- **`src/livekit_sales_agent/`** — Core library:
  - `config.py` — Dataclass settings models (ChatModelSettings, SttModelSettings, TtsModelSettings, AgentProfileSettings)
  - `knowledge/` — Full RAG stack: SQLite repositories, ChromaDB vector store, text chunking, embedding, async indexing jobs
  - `conversation/` — Conversation and message persistence
  - `adapters/` — Custom STT/TTS providers (ByteDance Doubao)
  - `prompts.py` — System instruction builder
  - `voice.py` — STT/TTS provider builder functions

### Data Layer

All runtime data lives in `.data/kb/` (git-ignored):
- `app.db` — SQLite database (knowledge bases, products, profiles, conversations, jobs)
- `chroma/` — ChromaDB vector embeddings
- `files/` — Uploaded documents

### Key Patterns

**Database-driven configuration**: Agent profiles, LLM/STT/TTS model profiles, and KB assignments are all stored in SQLite, not hardcoded env vars. Frontend UI edits these; the agent loads them at session start.

**RAG injection**: On each user turn, `_inject_retrieval_context()` queries ChromaDB for relevant KB documents and inserts them into the system message before LLM inference.

**Session modes**: Conversations run in `voice` mode (full STT/TTS pipeline) or `text` mode (no audio). Set via LiveKit session metadata.

**Idle controller**: `IdleCallController` tracks user/agent state transitions and sends reminder messages or ends the call after configurable timeouts (`AgentProfileSettings.idle_timeout_seconds`).

**Async file indexing**: File uploads enqueue jobs in SQLite; a background worker chunks, embeds, and stores vectors in ChromaDB.

### Frontend Structure

- `frontend/app/(workspace)/` — Main app routes (conversations, KB management, settings, products, agents)
- `frontend/app/api/` — Server-side API routes (token generation, KB/chat proxies)
- `frontend/components/app/` — Business logic components
- `frontend/components/agents-ui/` — Shadcn Agents UI components (installed via `pnpm shadcn:install`)
- `frontend/components/ui/` — Primitive shadcn/ui components

## Environment Variables

**Backend (`.env`)**: `LIVEKIT_URL`, `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET`, `AGENT_NAME`, `KB_DIR`, `KB_DATA_DIR`, `KB_TOP_K`, `PRICE_STALE_AFTER_DAYS`, `STT_FALLBACK_PROFILE_IDS`, `TTS_FALLBACK_PROFILE_IDS`

**Frontend (`frontend/.env.local`)**: Same LiveKit credentials + `NEXT_PUBLIC_DISPATCH_AGENT_NAME`, `KB_API_URL` (defaults to `http://127.0.0.1:8001`)

## Package Management

- Python: `uv` (with `UV_CACHE_DIR=.uv-cache` convention in scripts)
- Frontend: `pnpm` (v9.15.9 specified in `package.json`)
- pytest path is configured to `src` in `pyproject.toml`, so imports work without install
